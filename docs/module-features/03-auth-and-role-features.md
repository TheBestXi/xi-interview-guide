# 用户体系与角色权限功能设计文档

> 配合 `03-auth-and-role-design.md`（开发技术文档）使用。
> 本文档从**产品/用户体验视角**定义用户体系：用户能做什么、看到什么、流程怎么走、边界在哪。
> 技术实现细节（表结构、Redis Key、Lua 脚本、JWT 签发）见开发文档。

---

## 1. 模块定位

用户体系是整个产品的**地基**。它解决三件事：

1. **谁能用**：游客免登录体验（降低门槛）+ 注册用户完整功能（沉淀价值）。
2. **谁是谁**：用 deviceId（游客）或 JWT（登录）识别身份，数据不会乱串。
3. **谁干什么**：求职者 / 导师 / 管理员三类角色，权限分层。

它**不直接产生业务价值**，但所有业务模块（简历、面试、知识库）都挂在它上面。

---

## 2. 三类角色与权限矩阵

### 2.1 角色定义

| 角色 | 谁是 | 核心能力 |
|------|------|---------|
| **STUDENT**（求职者） | 注册用户默认角色 | 简历分析、模拟面试、知识库、日程——全部业务功能 |
| **MENTOR**（导师） | 手动配置/邀请 | 拥有 STUDENT 全部权限 + 可切求职者视角体验 + 导师专属功能（后定） |
| **ADMIN**（管理员） | 占位 | 第一版无路由、无后台页面，仅 `@RequireRole(ADMIN)` 注解占位 |

### 2.2 游客 vs 登录用户的权限对比

| 功能 | 游客 | 登录用户 |
|------|------|---------|
| 简历上传分析 | ✅ 每天 3 次 | ✅ 放宽 |
| 简历报告查看 | ⚠️ 半遮面（评分 + 前 3 条建议） | ✅ 完整报告 |
| 文字模拟面试 | ❌ → 弹登录 | ✅ |
| 语音面试 | ❌ → 弹登录 | ✅ |
| 知识库上传/管理 | ❌ → 弹登录 | ✅ |
| 知识库问答 | ❌ → 弹登录 | ✅ |
| 面试日程管理 | ❌ → 弹登录 | ✅ |
| PDF 报告导出 | ❌ → 弹登录 | ✅ |
| 数据保留 | 12 小时 | 永久 |

**核心转化诱饵**：完整简历报告、文字/语音面试、知识库、日程、PDF 导出——全部设为登录才能用。

### 2.3 导师切换求职者视角

- **纯 UI 切换**：导师账号还是导师，只是前端界面切换成求职者视图
- **数据归属**：导师以求职者视角产生的数据（传的简历、做的面试）**算导师自己的**
- **记忆**：切到求职者视角后，刷新/重开浏览器仍保持（存 localStorage）
- **后端约定**：求职者页面的 API 同时接受 MENTOR 和 STUDENT 角色，靠"已登录"拦截，不靠角色拦截

## 3. 游客体验流（核心引流路径）

### 3.1 全流程

```
用户首次访问
   │
   ├─ 前端自动生成 deviceId，存浏览器（用户无感）
   │
   ├─ 浏览产品（首页、介绍页）
   │     └─ 不建账号，不算配额
   │
   ├─ 点击"免费分析简历" → 上传简历（首次业务动作）
   │     └─ 后端此时才创建 Guest 账号
   │     └─ 消耗 1 次配额（今天剩 2 次）
   │
   ├─ 看到半遮面报告（评分 + 3 条建议）
   │     └─ "注册查看完整报告" CTA
   │
   ├─ 继续上传？第 4 次 → 配额耗尽
   │     └─ 弹注册框："今日免费体验已用完"
   │
   └─ 用户决定注册
         ├─ 填手机号 + 密码
         ├─ 注册请求自动带 reportId
         └─ 注册成功 → 报告自动补全 → 无缝接住完整内容
```

### 3.2 游客配额体验

| 场景 | 用户看到 |
|------|---------|
| 首次上传 | "剩余 2 次免费体验" |
| 第二次上传 | "剩余 1 次免费体验" |
| 第三次上传 | "剩余 0 次，注册后无限使用" |
| 第四次上传 | 配额耗尽，弹注册框 |

配额每天 0 点重置（Redis key 自动过期），次日用户回来又有 3 次。

## 4. 注册流程

### 4.1 流程

```
游客点"注册" CTA
   │
   ├─ 弹出注册表单
   │     ├─ 手机号
   │     ├─ 密码（输两遍确认）
   │     └─ 昵称（可选，默认用"游客_xxxx"）
   │
   ├─ 提交（自动带上当前 reportId，如果有）
   │
   ├─ 后端处理
   │     ├─ 把 Guest 账号原地升级（user_id 不变）
   │     ├─ 从 Redis 取完整报告，回填数据库
   │     ├─ 删 Redis 那份
   │     └─ 发 JWT
   │
   └─ 注册成功
         ├─ 页面无感刷新
         ├─ 之前的半遮面报告自动补全为完整版
         └─ 用户惊喜："我刚传的简历直接接上了"
```

### 4.2 表单字段

| 字段 | 必填 | 校验 |
|------|------|------|
| 手机号 | ✅ | 格式校验；唯一性校验（数据库拦截） |
| 密码 | ✅ | 长度 ≥ 8；输两遍确认 |
| 昵称 | ❌ | 默认"游客_xxxx"（沿用游客时的昵称） |

### 4.3 错误状态

| 错误 | 提示 |
|------|------|
| 手机号格式错 | "请输入正确的手机号" |
| 手机号已注册 | "该手机号已注册，直接登录？" |
| 网络失败 | "注册失败，请重试" |
| 注册成功但报告已过期 | "注册成功！您之前的报告已过期，请重新上传" |

## 5. 登录流程

### 5.1 流程

```
用户点"登录"
   │
   ├─ 弹出登录表单（手机号 + 密码）
   │
   ├─ 提交
   │
   ├─ 后端校验
   │     ├─ 手机号存在？
   │     ├─ 密码匹配？（BCrypt 比对）
   │     └─ 通过 → 发 JWT
   │
   └─ 登录成功
         ├─ 前端存 JWT 到 localStorage
         └─ 刷新页面，恢复登录态
```

### 5.2 登录方式

**唯一**：手机号 + 密码。

**砍掉的**：
- 验证码登录（学习项目，不引短信通道）
- 第三方登录（第一版不做）
- 邮箱登录（无邮箱字段）

### 5.3 登录安全

- 登录接口限流：同一 IP 每 10 分钟最多 10 次（防撞库）
- 密码 BCrypt 加密存储，不明文
- JWT 失效前，用户保持登录

### 5.4 错误状态

| 错误 | 提示 |
|------|------|
| 手机号不存在 | "该手机号未注册" |
| 密码错误 | "手机号或密码错误"（不暴露是哪个错，防探测） |
| 触发限流 | "操作过于频繁，请稍后再试" |

## 6. 修改密码

### 6.1 流程

```
已登录用户 → 个人中心 → 修改密码
   │
   ├─ 输入：旧密码 + 新密码（输两遍）
   │
   ├─ 后端校验旧密码
   │
   ├─ 通过 → 更新密码（BCrypt 加密）
   │
   └─ 成功
         └─ 是否强制重登？(待定，见 §10)
```

### 6.2 关于"忘记密码"

**第一版不做。** 理由：

- 忘记密码需要"证明你是号主" → 标准做法是发短信验证码
- 学习项目砍掉短信通道 → 没有验证码 → 忘记密码不安全
- 任何人知道手机号就能重置密码，不可接受

**替代方案**：用户忘记密码 → 引导重新注册一个新账号。

> 将来上正式短信通道时，再补"忘记密码"接口，扩展位已预留。

## 7. 数据保留与清理

### 7.1 用户感知到的数据生命周期

| 角色 | 数据保留 |
|------|---------|
| 游客 | 12 小时（上传简历后） |
| 登录用户 | 永久 |

### 7.2 游客数据丢失场景（用户需感知）

| 场景 | 后果 | 提示 |
|------|------|------|
| 上传后 12 小时内注册 | ✅ 报告自动继承 | 无感 |
| 上传后超 12 小时才注册 | ❌ 报告已过期 | "报告已过期，请重新上传" |
| 清浏览器缓存 | ❌ deviceId 重置 | 无提示（用户重新上传即可） |
| 换浏览器/设备 | ❌ 数据不互通 | 无提示（接受此局限） |

### 7.3 后端定时清理（用户无感）

每天凌晨 3 点，定时任务自动清理：
- 过期的游客简历数据（12 小时前的）
- 7 天未转正的孤儿 Guest 账号

用户完全无感，不影响体验。

## 8. 与各业务模块的联动

用户体系是地基，各业务模块挂在它上面：

| 业务模块 | 依赖点 |
|---------|--------|
| 简历分析 | 游客配额、半遮面报告绑定（核心引流点） |
| 模拟面试 | 锁登录（仅注册用户可用） |
| 语音面试 | 锁登录 |
| 知识库 | 锁登录 |
| 面试日程 | 锁登录 |
| PDF 导出 | 锁登录 |

**简历模块是唯一对游客开放的入口**，其他模块全部锁登录，通过简历半遮面引流注册。

## 9. 用户体验细节

### 9.1 登录态的获取与丢失

| 场景 | 行为 |
|------|------|
| 首次访问 | 无登录态，自动作为游客 |
| 注册成功 | 获取 JWT，变登录态 |
| 登录成功 | 获取 JWT，变登录态 |
| 关闭浏览器再打开 | 仍有登录态（JWT 在 localStorage） |
| 清浏览器缓存 | 登录态丢失，变回游客 |
| JWT 过期 | 变回游客（待定：静默刷新 vs 强制重登） |

### 9.2 跨设备/浏览器

- **不互通**：用户在 Chrome 注册的账号，在 Firefox 打开是游客
- **可登录**：用户在 Firefox 手动登录同一账号，能拿到自己的数据
- **游客数据不跨设备**：Chrome 传的简历，Firefox 看不到

这是学习项目接受的局限，不专门处理。

### 9.3 配额提示的展示时机

| 时机 | 提示 |
|------|------|
| 上传简历成功后 | "剩余 X 次免费体验" |
| 配额 ≤ 1 时 | "仅剩 X 次，注册后无限使用"（加粗提醒） |
| 配额耗尽 | 弹注册框（不再允许上传） |
