# 简历模块功能设计文档

> 配合 `04-resume-module-dev.md` 使用。
> 本文档从**产品/用户体验视角**定义简历模块：用户能做什么、看到什么、边界在哪。
> 技术实现细节见开发文档。

---

## 1. 模块定位

简历模块是**用户进入产品的第一站**，也是**唯一对游客开放**的核心功能。

它承担两个使命：
1. **价值展示**：用户上传简历，30 秒内拿到专业点评，体验"AI 真能帮我"。
2. **引流注册**：点评报告"半遮面"——评分和前 3 条建议免费看，完整报告需注册。
   这是从"游客"到"注册用户"最核心的转化点。

---

## 2. 用户角色与权限

| 功能 | 游客 | 登录用户（STUDENT/MENTOR） |
|------|------|--------------------------|
| 上传简历 | ✅ 每天 3 次 | ✅ 放宽（10-20 次/天） |
| 查看点评报告 | ⚠️ 半遮面（评分 + 前 3 条建议） | ✅ 完整报告 |
| 历史记录 | ❌ 锁登录 | ✅ |
| 导出 PDF | ❌ 锁登录 | ✅ 每天 10 次 |
| 简历版本对比 | ❌ 第一版不做 | ❌ 第一版不做 |

---

## 3. 核心功能流程

### 3.1 游客体验流（核心转化路径）

```
游客访问 → 首页看到"免费分析简历" CTA
   │
   ├─ 上传 PDF/Word（拦截图片，友好提示）
   │
   ├─ 等待 6-16 秒（异步处理，前端轮询）
   │     页面展示："正在解析您的简历..."
   │
   ├─ 报告页（半遮面）
   │     ├─ 评分大数字（例：75/100）
   │     ├─ 评分维度（项目深度/技能匹配/内容完整度/结构/表达）
   │     ├─ 前 3 条建议（高优先级，可操作）
   │     ├─ 【模糊遮罩】剩余 N 条建议 + 重写范例 + 详细点评
   │     └─ "注册查看完整报告" CTA（醒目）
   │
   ├─ 用户点 CTA → 注册弹窗（手机号 + 密码）
   │     └─ 注册请求自动带上 reportId
   │
   └─ 注册成功
         ├─ 报告自动补全：从 3 条 → 完整建议 + rewrite 范例
         ├─ 页面无感刷新，展示完整报告
         └─ 用户惊喜："我刚传的简历直接接上了"
```

### 3.2 登录用户体验流

```
登录用户上传简历
   │
   ├─ 等待解析（同上）
   │
   ├─ 报告页（完整）
   │     ├─ 评分 + 五维雷达图
   │     ├─ 全部建议（按类别/优先级）
   │     ├─ 重写范例（2-3 条，原句 vs 优化句）
   │     └─ "导出 PDF" 按钮
   │
   └─ 历史记录页
         └─ 查看过往所有简历报告
```

## 4. 报告内容详解

### 4.1 评分体系（沿用源项目，百分制）

| 维度 | 分值 | 评什么 |
|------|------|--------|
| 项目技术深度 | 0-40 | 是否避开烂大街项目（博客/外卖），是否有量化产出 |
| 技能匹配度 | 0-20 | 技术栈专业度，核心能力是否突出 |
| 内容完整性 | 0-15 | 模块顺序是否合理 |
| 结构清晰度 | 0-15 | 技术名词大小写规范 |
| 表达专业性 | 0-10 | 语言简洁度，是否有不专业词汇 |
| **总分** | **0-100** | — |

### 4.2 建议列表

每条建议包含：
- **类别**：内容/格式/技能/项目
- **优先级**：高/中/低
- **问题描述**：具体指出哪里有问题
- **改进建议**：可操作的修改方向

游客看到前 3 条（按优先级排序，优先展示高优先级），完整列表需注册。

### 4.3 重写范例（亮点功能）

从简历中挑选 2-3 条核心项目描述，基于 STAR 法则重写：

```
原句：
  负责电商订单系统的开发

重写后（STAR + 量化）：
  主导电商订单系统重构，基于 Spring Cloud 微服务架构，
  日均处理百万订单，通过引入 Redis 分布式锁解决超卖问题，
  QPS 从 800 提升至 5000
```

这是用户感知最强的功能——"AI 帮我把简历改得更好"。

### 4.4 排版混乱提示（兜底）

如果检测到简历排版可能有问题（多栏检测漏判，或 LLM 读取别扭）：
- 报告顶部展示温和提示："检测到您的简历排版较复杂，部分信息可能识别不全，建议上传纯文本版获得更准点评"
- 不阻断流程，点评照常给（但可能不准）

## 5. 上传约束

| 约束 | 值 | 说明 |
|------|-----|------|
| 支持格式 | PDF, DOC, DOCX | 图片格式前端拦截 |
| 文件大小 | ≤ 5MB | 超出提示"文件过大，请精简后上传" |
| 页数 | 建议 1-2 页 | 超过 3 页提示"简历建议精简，但可继续" |
| 每日上传（游客） | 3 次 | 超出弹注册 |
| 每日上传（登录） | 10-20 次 | 具体值待定 |
| 每日导出（登录） | 10 次 | 防 CPU 滥用 |

---

## 5.6 用户体验细节

### 6.1 上传中

- 拖拽 + 点击两种上传方式
- 上传后立即显示文件名 + 进度条
- 解析中显示骨架屏（避免白屏焦虑）
- 超过 20 秒未返回，提示"解析较慢，可稍后查看"

### 6.2 报告页

- **评分大数字**：居中，醒目（例：75 大数字 + "你的简历击败了 60% 的用户"）
- **五维雷达图**：直观展示短板（项目深度弱？技能匹配差？）
- **建议卡片**：按优先级排序，高优先级红色标记
- **半遮面遮罩**：模糊效果 + 锁图标 + "注册解锁" 按钮
- **CTA 设计**：颜色醒目，文案"免费注册，立即解锁完整报告"

### 6.3 错误状态

| 错误 | 提示 |
|------|------|
| 上传图片格式 | "暂不支持图片格式，请上传 PDF 或 Word 版本" |
| 文件过大 | "文件超过 5MB，请精简后上传" |
| 解析失败 | "解析失败，请检查文件是否损坏，或重试" |
| 配额用尽 | "今日免费体验已用完，注册后可继续使用"（弹注册框） |
| 排版混乱 | "检测到复杂排版，部分信息可能识别不全，建议上传纯文本版" |

## 7. 版本管理（第一版不做，记录决策）

### 7.1 第一版决策：不做版本对比

- 多次上传保留**评分历史**（用户看分数变化），但不存档全文
- 不做版本 diff（"v1 vs v2 差在哪"），技术复杂，收益低
- 理由：用户更关心"现在这份怎么样"，不是"跟上份差多少"

### 7.2 历史记录保留策略

- 登录用户：保留**最近 5 份**简历报告（超出自动删旧的）
- 游客：不保留历史（数据只留 12 小时，注册后绑定当前那一份）

## 8. 报告导出

### 8.1 定位

导出 PDF 是给导师/学员线下场景用的——"面试报告能不能导出 PDF 给导师看"（甲方开场提的需求）。

### 8.2 规则

- **仅登录用户**可导出（游客锁登录）
- 每天 10 次（防 CPU 滥用）
- 导出的 PDF 内容 = App 内完整报告（评分 + 雷达图 + 建议 + rewrite）

### 8.3 导出内容

- 封面：姓名 + 总分 + 日期
- 评分页：五维雷达图 + 分项得分
- 建议页：完整建议列表
- 重写页：原句 vs 优化句对比

## 9. 与登录模块的联动

简历模块是登录模块的**主要引流入口**。联动点：

| 联动点 | 说明 |
|--------|------|
| 半遮面 CTA | "注册查看完整报告" → 弹注册框，自动带 reportId |
| 配额耗尽 | 游客 3 次用完 → 弹注册框 |
| 注册绑定 | 游客注册时自动绑定当前报告，完整内容无缝接住 |
| 转正后数据 | resume_analysis 的 expire_at 置 NULL，报告变永久 |
| 导出锁登录 | 导出 PDF 必须 JWT 校验通过 |
