# 对话式面试引擎 · 功能设计文档

> 本文档从**产品/用户体验视角**定义对话式模拟面试。
> 技术实现见 `05-interview-conversational-dev.md`。
> 跟源项目（问卷式）的差异见本文档 §10。

---

## 1. 模块定位

**一句话**：让 AI 当真实面试官，跟用户多轮对话，动态追问、实时纠偏、按能力维度评分。

### 1.1 跟源项目的本质差异

源项目的"文字面试"是**问卷**：一次性出 N 道题，用户逐题作答，最后 AI 评分。问题预设、无追问、无上下文。

本模块是**对话**：AI 实时跟用户聊天，根据回答决定追问、切换话题、给提示、结束面试。这是 2024 年后 AI 面试的主流方向。

### 1.2 为什么改造

| 场景 | 源项目（问卷式）| 本模块（对话式）|
|------|---------------|---------------|
| 用户答完一题 | 自动跳下一题 | AI 评估回答，决定追问/切换/给提示 |
| 用户答得含糊 | 不管，下一题 | AI 追问"能展开吗" |
| 用户答错方向 | 不管，下一题 | AI 拉回主题"我们先回到 X" |
| 用户答得好 | 不管，下一题 | AI 深挖"那底层原理是？" |
| 面试官风格 | 冷冰冰出题 | 有人格、有节奏、有反馈 |
| 评分 | 按每题打分 | 按 Redis/MySQL 等能力维度打分 |

**核心价值**：更接近真实面试体验，用户练的不是"答题"，是"对话能力"。

## 3. 核心用户旅程

### 3.1 完整流程

```
1. 选方向
   用户从 11 个 Skill 方向中选一个（java-backend / ali-backend / frontend ...）
   可选：上传简历（基于简历个性化出题）

2. 配置面试
   - 难度：junior / mid / senior
   - 总轮数：默认 15 轮（可调 5-30）
   - 总时长：默认 30 分钟（可调 10-60）
   - [可选] 上传 JD（AI 按 JD 调整考点权重）

3. 开始面试
   AI 自动生成开场白（基于简历/Skill/JD）
   例："你好，看到你简历里提到电商订单系统，能先讲讲这个项目的架构吗？"

4. 多轮对话（核心）
   ├─ 用户回答
   ├─ AI 思考（状态评估，~1-2 秒）
   ├─ AI 流式输出回复（打字机效果）
   ├─ AI 可能的动作：深挖/切换/提示/结束
   └─ 循环，直到结束条件触发

5. 结束面试
   触发条件之一：
     - 所有核心考点覆盖完
     - 总轮数达到上限
     - 总时长达到上限
     - 用户主动"结束面试"

6. 生成报告（异步，~10-30 秒）
   用户在"报告生成中"页面等待
   完成后展示：总分 + 各维度得分 + 优势 + 改进点

7. 查看报告
   - 总分大数字
   - 各能力维度雷达图
   - 每个维度的对话片段引用（证据）
   - 导出 PDF

8. 历史记录
   所有过往面试的报告列表
   支持对比两场面试的进步
```

### 3.2 用户视角的状态显示

用户不需要看到内部状态机，但需要感知"AI 在干什么"：

| 内部状态 | 用户看到 |
|---------|---------|
| EVALUATING（状态评估中）| "AI 正在思考..."（loading 1-2 秒）|
| RESPONDING（生成回复）| 流式打字机输出 |
| COMPLETED（面试结束）| "面试结束，正在生成报告..." |
| EVALUATED（报告就绪）| 跳转报告页 |

## 4. AI 面试官的人格设计

这是对话式跟问卷式的核心差异——AI 要像真人面试官，不像出题机器。

### 4.1 人格特征

```
专业：
  - 一次只问一个问题
  - 问题清晰、有目的
  - 不绕弯子

礼貌但有挑战性：
  - 用户答对："不错，那我们再深入..."
  - 用户答错：不直接说错，先追问"你这么想的依据是？"
  - 用户卡住：给方向性提示，不直接给答案

自然：
  - 用口语化表达，不堆专业术语
  - 不用 Markdown 列表
  - 每轮回复不超过 3 句话
  - 偶尔有"嗯/好的/明白"这种过渡词

有节奏感：
  - 不在同一考点死磕（最多 4 轮）
  - 主动切换话题，覆盖面广
  - 时间快到了会提醒
```

### 4.2 不该出现的行为

```
❌ "你的回答是错误的。正确答案是..."  ← 直接给答案，没教育意义
❌ "好的，下一题：..."                ← 冷冰冰，像出题机器
❌ "Redis 的 RDB 是 xxx，AOF 是 xxx，区别是 xxx" ← 一次说太多，喧宾夺主
❌ "**Redis 持久化**\n- RDB: ...\n- AOF: ..."  ← 用 Markdown，不像对话
❌ 一次问 3 个问题："RDB 是什么？AOF 是什么？区别是什么？"  ← 用户懵
```

## 5. 对话场景示例

### 场景 1：用户答得好，AI 深挖

```
AI：你简历里提到用 Redis 做排行榜，能讲讲为什么选 Redis 吗？
用户：因为 Redis 是内存数据库，读写快，而且 ZSet 天生支持排序。
AI：不错。那 ZSet 底层用了什么数据结构？
用户：跳表和压缩列表。
AI：为什么用跳表而不是红黑树？         ← 深挖底层
用户：跳表实现简单，范围查询效率好。
AI：范围查询具体是怎么实现的？           ← 继续深挖
用户：呃...通过 level 数组...
AI：（给提示）想想看，跳表的层级结构怎么帮助范围查找？  ← GIVE_HINT
```

### 场景 2：用户答不动，AI 切换话题

```
AI：MySQL 的索引底层是什么结构？
用户：B+ 树。
AI：B+ 树跟 B 树的区别是什么？
用户：...不太清楚。
AI：没关系，我们换个方向。你用过 Spring 的什么模块？  ← SWITCH_TOPIC
```

### 场景 3：用户答偏了，AI 拉回

```
AI：你提到分布式锁，你们怎么实现的？
用户：我们用的是 Kafka 做消息队列，吞吐量很高...
AI：（打断）明白你们用了 Kafka。我们先回到分布式锁——你们用的是 Redis 还是 Zookeeper？  ← 拉回主题
```

### 场景 4：时间到，AI 收尾

```
AI：好的，今天的面试就到这里。你 Redis 底层掌握得不错，分布式事务可以再深入。
    我们会生成详细的评估报告，稍等片刻。
→ 触发维度评估（异步）
```

## 6. 评分报告

### 6.1 评分维度（不按题打分）

报告不展示"第 1 题 80 分"，展示"Redis 85 分"。

```
总分：72/100

能力维度雷达图：
  Redis        ████░ 85  EXCELLENT
  MySQL        ███░░ 60  AVERAGE
  Spring       ████░ 78  GOOD
  分布式       ██░░░ 45  WEAK
  JVM          ███░░ 65  AVERAGE

每个维度点击展开：
  - 维度评分依据（用户原话引用）
  - 优势点
  - 改进建议
```

### 6.2 报告内容详解

| 板块 | 内容 |
|------|------|
| 总分 | 0-100，各维度加权平均（CORE 权重高）|
| 维度雷达图 | 直观展示强弱 |
| 维度详情 | 每个维度的对话片段 + 评价 |
| 优势 | 3-5 条（"Redis 底层原理掌握扎实"）|
| 改进点 | 3-5 条（"分布式事务场景理解偏浅"）|
| 对话全文 | 完整面试记录（可展开/折叠）|

### 6.3 报告示例

```
==========================================
面试报告 - Java 后端（中级）
==========================================

总分：72/100  (击败 65% 的同方向求职者)

▎能力维度
  Redis        ████████░░ 85  优秀
  MySQL        ██████░░░░ 60  中等
  Spring       ████████░░ 78  良好
  分布式       ████░░░░░░ 45  薄弱
  JVM          ██████░░░░ 65  中等

▎Redis（85 分 - 优秀）
  证据 1：用户准确说出 ZSet 用跳表实现，并能讲清范围查询优势。
        AI 问："为什么用跳表不用红黑树？"
        用户："跳表实现简单，范围查询效率好。"
  证据 2：用户对 AOF 的 fsync 策略理解到位。
  
▎分布式（45 分 - 薄弱）
  证据 1：用户混淆了分布式锁和分布式事务。
        AI 问："分布式事务怎么处理？"
        用户：答非所问。
  改进建议：补强 Seata / 消息事务 / TCC 等方案。

▎优势
  • Redis 底层数据结构理解深入
  • Spring IOC/AOP 原理清晰
  
▎改进点
  • 分布式事务场景理解偏浅，建议补强
  • JVM 调优经验不足，建议实践
```

## 7. 跟简历联动

### 7.1 简历驱动的个性化出题

上传简历后，AI 开场和考点权重会个性化：

```
无简历：
  AI 开场："你好，我们开始 Java 后端面试。先聊聊你对 Spring 的理解。"
  考点权重：标准分布（Skill 定义）

有简历：
  AI 开场："看到你简历里提到电商订单系统，能先讲讲这个项目的架构吗？"
  考点权重：
    - 简历提到的技术栈权重提高（Redis/Spring Cloud）
    - 简历没提的考点降低权重（但 CORE 仍必考）
```

### 7.2 简历作为评分参考

评估时，AI 拿简历当参考——简历写了"精通 Redis"，但实际答得不行，扣分更狠（"简历造假"）。

## 8. 用户体验细节

### 8.1 流式输出体验

AI 回复用流式（SSE），打字机效果，降低等待焦虑：

```
用户提交回答
  → "AI 思考中..."（1-2 秒，状态评估）
  → "不错"（开始流式）
  → "不错，那 ZSet"
  → "不错，那 ZSet 底层用了"
  → "不错，那 ZSet 底层用了什么数据结构？"
  → 完成
```

### 8.2 错误状态

| 错误 | 处理 |
|------|------|
| AI 回复中断 | 自动重试 1 次，失败提示"网络异常，请重试" |
| 评估超时 | 提示"报告生成较慢，可稍后查看" |
| 用户中途退出 | 状态保存，可继续（24h 内）|
| 用户刷新页面 | 从最近状态恢复（Redis 缓存）|

### 8.3 面试节奏控制

```
轮数提示：
  "第 5/15 轮"  → 进度条

时长提示：
  剩余 5 分钟时，AI 主动说："时间不多了，我们再聊一个考点"
  时间到，AI 自动收尾

用户主动控制：
  - "暂停面试"：保存状态，可继续
  - "结束面试"：立即触发评估
  - "下一题"：跳过当前考点（不计分）
```

## 9. 跟源项目的差异对照

| 维度 | 源项目（问卷式）| 本模块（对话式）|
|------|---------------|---------------|
| 产品形态 | 问卷 | 对话 |
| AI 角色 | 出题机器 | 真人面试官 |
| 追问 | 预设 | 动态 |
| 上下文 | 无 | 累积 |
| 评分 | 按题 | 按维度 |
| 报告 | 题目分数 | 维度雷达图 + 证据 |
| 体验 | 冷冰冰 | 有人格 |
| 成本 | ~0.05 元/场 | ~0.10-0.15 元/场 |
| 技术难度 | 中 | 高 |
